### Profile Data Model: MVP vs Future Vision

---

#### FOR FUTURE SESSIONS: THE FULL VISION

**IMPORTANT CONTEXT (December 2025 discussion):**

The user's vision for the profile system is MUCH bigger than what we're building for the Digest 2.0 MVP. Here's what was discussed and agreed:

**1. "ME AS ENTITY" ARCHITECTURE**
- The user should NOT be a separate "profile" system
- The user IS a special `person` entity in the SAME system that will hold all network contacts (Phase 3+)
- Same data model: timeline, events, context, connections, memories, highlights, successes, failures
- Think: diary + CV + relationship map - all queryable, growing over time
- The user just has MORE data and special privileges, not a different architecture
- This means: when we build the Network CRM (Phase 3), we're also building the "Me" system

**2. WRITING STYLE IS A FEATURE AREA, NOT A CONFIG FIELD**
- Cannot be captured in a few sentences
- Needs hierarchical layers:
  - BASE STYLE (general defaults)
  - CONTENT TYPE MODIFIERS (LinkedIn vs blog vs email vs newsletter)
  - SUB-TYPE MODIFIERS (LinkedIn study dissection vs LinkedIn contrarian take vs LinkedIn observation - totally different styles!)
  - PER-PIECE ITERATION (feedback loop for each draft)
- May need its own "Writing Style Engine"
- Should learn from accepted vs rejected drafts over time
- Example corpus of "posts I liked" as reference

**3. UNSTRUCTURED INPUT â†’ STRUCTURED EXTRACTION**
- User will add information in freeform way: "Had a conversation with Sarah about AI..."
- System extracts entities, events, connections, context
- Presents for confirmation before saving
- Everything becomes queryable and connected

**4. EVERYTHING IS A LIVING ORGANISM**
- Profile grows continuously
- More data = more value
- Connections form between entities, topics, events
- Query anything: "Show all AI-related context for me and my VIP contacts"

---

#### MVP APPROACH (Digest 2.0 - Now)

**What we build now (starter only):**
- Simple JSON blob for basic profile data
- Just enough for Belief Aligned/Challenger lenses
- Just enough for basic draft tone hints
- Clearly marked as "Basic" in UI with note "More depth coming soon"
- MUST be able to migrate forward to the full entity system

```sql
-- MVP: Single flexible JSON blob, easy to evolve
-- This is TEMPORARY - will be replaced by unified entity system in Phase 3
CREATE TABLE user_profile_mvp (
    id INTEGER PRIMARY KEY DEFAULT 1,
    profile_json TEXT,  -- All profile data as flexible JSON
    created_at TEXT,
    updated_at TEXT
);

-- Example profile_json structure (not schema-enforced, can add fields freely):
-- {
--   "beliefs": ["I value privacy over convenience", ...],
--   "interests": {"expert_in": [...], "interested_in": [...], "learning": [...]},
--   "style_hints": {"tone": "...", "avoid": [...], "prefer": [...]},
--   "background_notes": "Free text - CV highlights, anecdotes, etc."
-- }
```

**Why this approach:**
- Don't lock into rigid schema before entity system is designed
- JSON is flexible, can add fields without migrations
- Easy to migrate: extract from JSON â†’ insert into proper tables
- Full entity system design happens with Phase 3 (Network MVP)

## Revised Implementation Sequence

With your feedback, I'm reordering priorities:

### Phase A: Source Transparency (Foundation)
**Why first:** Everything else depends on traceable sources. Without this, curation and draft bridging have no meaning.

- Wire up `digest_citations` properly (currently unused)
- Every topic/insight links to source chunks with context
- Source drill-down UI (collapsible/modal)
- "View sources" on every element

### Phase B: Interactive Curation
**Why second:** This transforms digests from static reports into workspaces. Enables the draft bridge.

- Include/exclude toggles on topics, findings, quotes
- Star rating system (â˜…â˜…â˜…/â˜…â˜…â˜†/â˜…â˜†â˜†)
- Drag-to-reorder priority
- "Set as Main Story" / "Set as Lead Hook" markers
- Store curation state per digest

### Phase C: Draft Bridge
**Why third:** The payoff. Content transformation pipeline becomes real.

- "Create Draft from Selection" action
- Format selection (LinkedIn, Newsletter, Thread, Blog, Generic)
- Draft configuration (main story, hook, tone, angle)
- Draft inherits sources, quotes, findings
- Full provenance chain preserved

### Phase D: Pipeline Visibility
**Why fourth:** Improves the experience but not blocking core functionality.

- Enhanced SSE events with phase details
- Real-time cluster visualization
- Activity log
- Clear post-completion state

### Phase E: Lens System Foundation
**Why fifth:** Now that digests are useful, make them configurable.

- Built-in lens presets
- Lens selection in digest creation
- Store lens_id with digest
- "Try different lens" action

### Phase F: Lens Editor with Preview
**Why sixth:** Power user feature, needs the foundation first.

- Test corpus for previewing
- Side-by-side before/after
- Full prompt editing
- Clone/modify lenses

### Phase G: Belief-Aware Lenses
**Why last:** Requires user profile, depends on lens system.

- User profile/beliefs configuration
- "Belief Aligned" lens
- "Belief Challenger" lens
